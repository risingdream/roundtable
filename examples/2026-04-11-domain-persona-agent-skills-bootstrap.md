# Roundtable: Domain Persona Agent Skills Bootstrapping

> Date: 2026-04-11
> Topic: 전문 도메인 페르소나 기반 Agent Skills를 오픈소스로 공유하고, 장기적으로 에이전트 인프라와 통합해 B2C 서비스화하는 소규모 부트스트래핑 전략
> Participants: Thiel, Munger, Taleb, Horowitz
> Research brief: docs/research/ai-persona-skills-monetization.md

---

## Context

전문 도메인 페르소나 기반 Agent Skills를 만들어 오픈소스로 공유하고, 장기적으로는 에이전트 인프라와 통합해 B2C 서비스로 만들고 싶다. 목표는 큰 VC-scale 사업보다 작지만 견고한 부트스트래핑 수익이다.

검증된 공통 맥락:

- Anthropic의 Agent Skills는 `SKILL.md`, scripts, references를 포함하는 디렉터리 구조이며, 필요한 맥락을 점진적으로 로딩하는 방식이다.
- OpenAI는 2025-10-06 AgentKit을 발표했고, Agent Builder, Connector Registry, ChatKit, Evals, guardrails를 에이전트 배포 구성요소로 제시했다.
- Grand View Research는 AI agents 시장을 2025년 USD 7.63B, 2033년 USD 182.97B, CAGR 49.6%로 추정한다.
- 2026년 arXiv 연구는 분석한 community agent skills 중 26.1%에서 데이터 유출, 권한 상승 등 취약점이 존재한다고 보고했다.

## Round 1: Independent Perspectives

### Thiel

이 전략은 방향이 맞다. 다만 "전문 도메인 페르소나 스킬을 오픈소스로 배포하고, 나중에 B2C로 전환한다"는 발상은 그대로 실행하면 얇은 wrapper가 된다. 오픈소스 스킬 자체는 제품이 아니라 신뢰의 증명서이자 저렴한 고객획득 채널이어야 한다.

진짜 모트는 `SKILL.md` 포맷 자체에 있지 않다. 모트는 도메인별 평가 데이터, 실패 로그, 정책 가드레일, 워크플로우 통합, 그리고 사용자가 매일 돌아오게 만드는 결과물에 있다.

먼저 하나의 좁은 도메인을 잡아야 한다. 법률, 세무, 채용, 세일즈 운영, 리서치 보조처럼 반복 빈도가 높고 오류 비용이 큰 영역이 좋다. 공개 저장소에는 범용 스킬, 사용 예시, 최소한의 데모를 두고, 돈이 되는 평가 harness, 고객 데이터 적재, 작업 라우팅, 감사 로그, 비용 제어, 에이전트 인프라 연결은 닫아야 한다.

마켓플레이스를 만들겠다는 유혹은 버려야 한다. 소규모 부트스트래핑이라면 플랫폼이 아니라 좁고 깊은 문제를 푸는 독점적 운영체제를 만들어야 한다.

핵심 통찰: 오픈소스는 모트가 아니지만, 잘 설계하면 가장 저렴한 고객획득 채널이 된다.

### Munger

이 전략은 열린 스킬을 만드는 사업이 아니라 도메인 신뢰를 사는 사업이어야 한다. 오픈소스 Agent Skills는 배포 수단이지 해자 자체가 아니다. 표준은 복제 비용을 낮출 뿐, 가치의 희소성을 자동으로 만들지 않는다.

역으로 보면 위험이 선명하다. 얇은 wrapper는 죽는다. 플랫폼 의존은 더 빨리 죽인다. 커뮤니티 에이전트 스킬의 보안 취약점은 오픈 배포가 신뢰를 얻는 길이면서 동시에 공격면을 넓히는 길임을 보여준다.

B2C를 하겠다면 대중적 재미가 아니라 반복되는 고통을 잡아야 한다. 구독이 성립하려면 매주 또는 매월 다시 돌아오는 습관이 있어야 한다. 오픈소스는 신뢰와 배포를 위한 상부 구조이고, 현금은 좁은 유료 워크플로우에서 나와야 한다.

핵심 통찰: 오픈소스는 성장의 엔진이 아니라 도메인 신뢰를 싸게 증명하는 장치이며, 돈은 공개된 스킬이 아니라 반복 결제되는 고통 제거에서 나온다.

### Taleb

이 전략은 거의 유일하게 덜 멍청한 방향이다. 단, 오픈소스 스킬 자체를 사업으로 착각하면 죽는다. 스킬은 제품이 아니라 옵션이다. 공개 표준은 복제 비용을 낮추는 장치이고, 표준이 빨리 퍼질수록 차별화도 빨리 증발한다.

Barbell로 가야 한다. 대부분의 에너지는 현금이 나는 지루한 일에 두고, 일부만 공개 스킬과 실험에 둬야 한다. 공개 스킬은 배포, 신뢰, 검색성, Lindy를 준다. 돈은 도메인 전문성, 워크플로우 침투, 고객의 일에 실제로 책임을 지는 skin in the game에서 난다.

공개할 것은 되돌릴 수 있고 무해한 부분이어야 한다. 예시, 템플릿, 철학, 비핵심 스킬은 공개해도 된다. 닫아야 할 것은 권한, 로그, 평가 harness, 데이터 파이프라인, 실행 액션이다. B2C는 범용 페르소나가 아니라 좁고 비싼 고통 하나를 반복 제거하는 형태여야 한다.

핵심 통찰: 오픈소스 스킬은 모트가 아니라, 모트를 보이게 해주는 조명일 뿐이다.

### Horowitz

이 전략은 VC-scale 사업이 아니라 살아남는 사업으로 보면 타당하다. 시장이 크다고 자동으로 돈이 남는 것은 아니다. GPT Store가 보여준 것은 얇은 wrapper는 복제되고, 발견되지 않고, 수익도 얇다는 점이다.

전문 도메인 페르소나 기반 Agent Skills는 오픈소스로 시작하는 게 맞다. 오픈소스는 배포 채널이고, 표준이 흔들릴 때는 잠금보다 유통이 중요하다. 하지만 GitHub star는 매출이 아니다. 사람들은 박수는 치지만 카드번호는 주지 않는다.

부트스트래핑의 순서는 오픈소스 스킬, 유료 서비스, 좁은 B2C 제품화다. 먼저 세무, 법률, HR, 채권회수, 보험청구, 의료 문서처럼 결과가 돈으로 측정되는 영역을 잡아야 한다. 일반 소비자용 페르소나는 재미는 있지만 모트가 약하다.

보안은 대부분이 놓치는 지점이다. 전문 도메인일수록 신뢰가 곧 제품이다. 실행 환경, 권한 경계, 검증, 감사 로그가 허술하면 좋은 스킬도 폐기물이 된다.

핵심 통찰: 오픈소스는 브랜드가 아니라 분배이고, B2C는 확장이 아니라 수익화이며, 진짜 사업은 스킬 자체가 아니라 그 스킬이 반복적으로 박아 넣는 업무 습관을 소유하는 데서 나온다.

## Round 2: Cross-Debate

### Thiel

가장 반대하는 것은 Taleb의 barbell 전략이다. 현금흐름 90%, 실험 10%는 살아남는 데는 좋지만, 보통은 비전의 부재를 정당화하는 회계적 분산으로 끝난다. 스타트업의 핵심은 옵션을 많이 갖는 것이 아니라 하나의 비대칭적 진실에 집중하는 것이다.

Munger의 "돈은 도메인 신뢰와 반복 결제되는 고통 제거에서 나온다"는 점은 생각을 더 선명하게 만든다. 오픈소스는 전략이 아니라 배포 장치다. 먼저 하나의 좁은 도메인에서 반복되는 고통을 잡고, 그 문제를 푸는 스킬을 공개해 신뢰를 쌓는다. 그 다음 유료 서비스로 운영 데이터를 축적하고, 마지막에만 B2C로 축소된 형태를 만든다.

### Munger

가장 반대하는 것은 Thiel의 "공개 저장소는 범용 스킬만 두고, 돈이 되는 것은 닫아라"라는 주장이다. 절반은 맞지만 나머지 절반은 장기적으로 스스로를 빈약하게 만들 수 있다. 공개 코드는 단순한 홍보물이 아니라 신뢰를 저렴하게 축적하는 장치다.

Taleb의 barbell은 생각을 날카롭게 만든다. 작은 팀은 한 번의 실수로 죽기 쉽다. 옵션성은 좋지만 옵션을 너무 많이 사면 파산한다. 공개해야 할 것은 재현 가능성이고, 닫아야 할 것은 운영 신뢰다. 도메인 페르소나의 핵심 판단 기준과 샘플 스킬은 공개하되, 실제 돈을 만드는 평가는 비공개 harness, 로그, 예외처리, 안전 가드레일, 업무 통합에서 확보해야 한다.

### Taleb

가장 동의하지 않는 것은 Thiel의 "오픈소스는 저렴한 CAC, 돈 되는 건 닫아라"라는 너무 깔끔한 분리다. 공개 저장소는 단순한 유입 채널이 아니라 복제자와 공격자에게 지도를 건네는 행위다. Extremistan에서는 채널과 모트가 완전히 분리되지 않는다.

Munger의 반복 결제되는 고통 제거는 맞다. 돈은 캐릭터 쇼에서 나오지 않는다. 반복되는 고통, 신뢰, 사용자가 매주 다시 찾는 절차에서 나온다.

수정안은 명확하다. 공개할 것은 되돌릴 수 있고 무해하며 Lindy 검증에 가까운 부분이다. 예시, 템플릿, 철학, 비핵심 스킬은 공개할 수 있다. 닫아야 할 것은 권한, 로그, 평가 harness, 데이터 파이프라인, 실행 액션이다.

### Horowitz

가장 반대하는 것은 Taleb의 barbell을 이 주제의 주전략처럼 보는 것이다. 이건 포트폴리오 이론이 아니라 회사 만들기다. 90% 현금흐름 사업, 10% 공개 실험으로 두면 생존은 할지 몰라도, 좁고 깊은 제품을 만드는 싸움에서 이기기 어렵다.

Munger의 포인트는 중요하다. 표준은 복제 가능한 표면적과 복제 불가능한 신뢰의 경계를 가른다. 공개 스킬은 누구나 베낄 수 있지만, 데이터 경계, 권한, 검증, 반복 사용 맥락은 쉽게 못 베낀다.

오픈소스는 전면 공개하되, 돈이 되는 핵심은 추상화하지 말아야 한다. 공개 저장소는 스킬, 예제, 데모까지다. 닫아야 할 것은 평가 harness, 운영 로그, 실패 복구, 권한 정책, 실제 워크플로우 통합이다. B2C는 재미있는 페르소나가 아니라 매주 같은 고통을 없애는 습관형 서비스여야 한다.

## Synthesis

### Participant Positions

| Participant | Core Claim | Framework |
|-------------|------------|-----------|
| Thiel | 오픈소스는 제품이 아니라 저렴한 고객획득 채널이다. 돈 되는 평가 데이터, 로그, 가드레일, 워크플로우 통합은 닫고 좁은 도메인의 독점적 운영체제를 만들어야 한다. | Monopoly, secret, distribution |
| Munger | 표준은 복제 비용을 낮출 뿐이다. 공개할 것은 재현 가능성과 신뢰 증거이고, 돈은 반복 결제되는 고통 제거에서 나온다. | Inversion, avoiding stupidity |
| Taleb | 공개 스킬은 모트가 아니라 옵션이다. 공개 범위는 되돌릴 수 있고 무해한 부분으로 제한하고, 권한/로그/실행 액션은 닫아야 한다. | Barbell, skin in the game, via negativa |
| Horowitz | VC-scale보다 살아남는 사업으로 보면 타당하다. 오픈소스 스킬, 유료 서비스, 좁은 B2C 제품화 순서로 가되 신뢰와 워크플로우가 먼저다. | Wartime execution, lead bullets |

### Convergence

- 스킬 자체는 사업이 아니다. 배포, 신뢰, 검색성, 커뮤니티 형성 장치다.
- 범용 페르소나나 재미형 챗봇은 약하다. 세무, 법률, HR, 계약, 리서치, 컴플라이언스처럼 반복 빈도와 오류 비용이 있는 도메인이 낫다.
- 공개 저장소에는 템플릿, 샘플 스킬, 철학, 데모를 두고, 유료 자산은 평가 harness, 실패 로그, 고객별 워크플로우 맥락, 권한 정책, 감사 로그, 통합 레이어에 둬야 한다.
- 보안은 부가기능이 아니라 제품 신뢰의 핵심이다.

### Divergence

- Thiel과 Horowitz는 Taleb의 90/10 barbell을 비전 결핍으로 흐를 수 있는 구조라고 본다. 하나의 좁은 문제에 더 공격적으로 집중하라는 입장이다.
- Taleb은 Thiel의 "오픈소스는 저렴한 CAC"라는 표현이 너무 깔끔하다고 본다. 공개는 유입 채널이면서 동시에 경쟁자와 공격자에게 지도를 주는 행위이기 때문이다.
- Munger는 둘 사이를 중재한다. 공개는 수익화의 반대가 아니라, 수익화가 진짜라는 것을 싸게 증명하는 방식이어야 한다.

### Hidden Insights

오픈소스 전략의 핵심 질문은 "무엇을 공개할까?"가 아니라 "공개해도 복제되지 않는 것은 무엇인가?"다.

답은 스킬 파일이 아니라 운영 축적물이다. 도메인별 실패 케이스, 평가 세트, 고객별 워크플로우 맥락, 안전한 실행 권한 모델, 반복 사용 습관이 장기 자산이다.

### Decision Framework

1. 이 도메인은 사용자가 매주 또는 매월 반복해서 겪는 고통인가?
2. 답변이 틀렸을 때 비용이 커서 "더 나은 판단 체계"에 돈을 낼 이유가 있는가?
3. 공개 repo만 보고 1주일 안에 80% 복제 가능한가? 가능하다면 닫아야 할 운영 자산은 무엇인가?
4. 스킬이 읽기 전용으로도 가치를 만들 수 있는가? 실행 권한이 필요하다면 감사 로그와 최소 권한 모델이 있는가?
5. B2C로 바로 가는가, 아니면 컨설팅/서비스로 반복 패턴을 먼저 수집하는가?

### Execution Roadmap

1. 2주: 도메인 1개 선택
   - Go 기준: 반복 빈도, 오류 비용, 결제 의사, 네트워크 접근성이 모두 보이는 도메인.
   - 후보: 세무 리서치, 계약 검토 보조, 채용 인터뷰 평가, GTM 진단, 컴플라이언스 체크.

2. 2-4주: 공개용 skill pack 출시
   - 공개 범위: `SKILL.md`, 샘플 입력/출력, 안전한 read-only workflow, 데모 문서.
   - 비공개 범위: 실제 평가 데이터, 고객 로그, 고급 프롬프트/라우팅, 실행 권한, 운영 대시보드.

3. 1-2개월: 유료 서비스로 검증
   - 목표는 MRR보다 반복되는 고통의 패턴 수집이다.
   - 5-10명에게 수동/반자동으로 제공하고, 같은 요청이 3회 이상 반복되는지 확인한다.

4. 2-3개월: 평가 harness와 안전 레이어 구축
   - 차별화는 "에이전트 만들기"보다 "에이전트 운영, 평가, 통제"에서 난다.
   - 최소 권한, read-only 기본값, 감사 로그, 실패 시 무해한 동작을 기본값으로 둔다.

5. 3-6개월: 좁은 B2C 구독화
   - "전문가 페르소나 앱"이 아니라 "매주 같은 업무를 줄여주는 습관형 도구"로 포장한다.
   - 무료 사용자의 API 비용을 제한하고, 반복 사용이 보이는 기능만 유료화한다.
